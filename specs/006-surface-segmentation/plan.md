# Implementation Plan: Venue Surface Segmentation & Spatial Layout

**Branch**: `006-surface-segmentation` | **Date**: 2026-07-04 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/006-surface-segmentation/spec.md`

## Summary

Surfaces gain first-class bookable segments authored by drawing zones on a surface schematic ("geometry proposes, declarations decide" — drawn geometry suggests symmetric coexistence relationships; staff confirm; stored declarations drive all conflict math). A single availability engine spans five booking sources (venue schedule blocks incl. recurrence expansion, signup-event games, season games, team calendar events, venue-attached practices) and replaces the fragmented per-subsystem checkers. The read-only venue-admin surface/hours/blocks screens from 002 become fully manageable, a schematic venue layout editor renders on public rink profiles, scheduling flows swap the display-only ice-usage enum + free-text zone labels for structured segment pickers (pre-launch, replaced outright), and practice sessions optionally attach to venue/surface/segment with a start time.

## Design Decisions (Phase 0/1 summary)

1. **Whole-surface = `segmentId: null` on bookings** — no autocreated WHOLE rows to backfill or protect; pickers render the whole-surface option from `IceSurface.wholeLabel ?? preset default` (satisfies "preset full is the renameable implicit whole-surface segment"). Conflict math: null segment conflicts with every booking on the surface.
2. **Coexistence = presence-only canonical pair rows** (`SegmentCoexistence` with `segmentAId < segmentBId`, unique) — row present ⇒ coexist; absent ⇒ conflict (safe default is free). Symmetry by construction (FR-005).
3. **Geometry is authoring input only** — normalized rects `{x,y,w,h,rotation}` stored per segment; `segment-geometry.ts` computes overlap → suggested pairs; actions persist only staff-confirmed pairs (FR-005). Re-drawing re-runs suggestions but never silently mutates declarations.
4. **Preset roles** — `presetRole` strings (`HALF_A`, `CROSS_2`, …) make re-application idempotent by role, not name (FR-004); presets defined per `SurfaceType` in `segment-presets.ts` (ice fully populated; court halves; others whole-only) with standard geometry + coexistence matrix (halves coexist; four cross zones mutually coexist and coexist with the opposite half; cross conflicts its containing half).
5. **One availability engine** — `availability.ts` exposes `findBookingConflicts(candidate)` over five sources: Events (venue-wide), SeasonGames (SCHEDULED/COMPLETED), EventGames (parent SignupEvent PUBLISHED, game not CANCELED), VenueScheduleBlocks (PUBLISHED, expanded via existing `expandRecurrenceWindow`), PracticeSessions (venue-attached with `startAt`). Same warn-then-recorded-override policy as 005 in every flow. `game-conflicts.ts` is absorbed and retired.
6. **EventGame + SeasonGame lose usage/zone fields; gain `segmentId`** — pre-launch replace-outright (FR-014); `IceUsage` enum removed; sport-catalog drops `surfaceUsageOptions` (segment presets by SurfaceType supersede it — SC-007 holds because non-ice surfaces get non-ice presets); 005's sport-awareness tests are updated to assert segment-picker behavior instead.
7. **Practice attachment** — nullable `venueId/surfaceId/segmentId/startAt` on PracticeSession; attaching a venue requires `startAt`; slot = `startAt + duration`; planner warns via the engine with the standard override (FR-019).
8. **Venue layout as `Venue.layout Json?`** — `{ surfaces: [{surfaceId,x,y,w,h,rotation}], labels: [{text,x,y}] }`, schematic only; editor warns on visual overlap; public map renders read-only (FR-016–018); saves recorded in VenueActivityLog.
9. **Finish 002's admin UI** — surface CRUD/hours actions exist with zero call sites; block CRUD UI is also missing and is required for segment-aware blocks (FR-013) and the schedule view (FR-021): SurfaceManager + VenueScheduleBoard close both gaps.
10. **Migration mechanics** — same as 005: author SQL via `prisma migrate diff --config`, apply to dev via `db:push`, regenerate client; schema + migration committed together.
